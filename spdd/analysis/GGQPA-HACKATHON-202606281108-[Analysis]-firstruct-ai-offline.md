# SPDD Analysis: FIRStruct AI (Offline FIR Generator)

## Original Business Requirement
The goal is to build FIRStruct AI, an offline-first, CPU-optimized AI application for police stations. It transforms unstructured handwritten or printed complaints into structured FIR drafts without internet access.
The pipeline involves: OpenCV for image preprocessing -> PaddleOCR for text extraction -> Text Cleaning -> Local LLM (Ollama) for entity extraction and JSON structuring -> SQLite for storage -> Streamlit for the dashboard.
It extracts details like Victim, Accused, Date, Time, Location, Vehicle, Weapon, Stolen Items, and suggested IPC/BNS sections.

#### New Concepts Required
- **Complaint Image**: The raw uploaded file (JPG, PNG, PDF) representing the handwritten or printed citizen complaint.
- **Raw Complaint Text**: The intermediate unstructured text generated by the OCR engine.
- **FIR Draft (Structured FIR)**: The core business entity containing the extracted legal fields (Victim, Accused, Crime Type, Date, Location, Vehicles, Weapons, Stolen Items, IPC Sections).
- **Extracted Entity**: Supporting relational concepts like `Vehicle`, `Weapon`, and `IPCSection` that allow for deep searching (e.g., "Find all bike thefts" or "Search knife attacks").

#### Key Business Rules
- **Absolute Privacy / Offline Execution**: Sensitive citizen and crime data must never leave the device. No cloud APIs can be used.
- **Human-in-the-Loop**: The system generates an FIR *draft*. An officer must review and approve it before finalization.
- **Data Structuring**: The LLM must output valid JSON matching the strict FIR schema to ensure the database remains searchable.

## Strategic Approach

#### Solution Direction
The application will be a Streamlit-based Python web app. The processing pipeline is highly sequential:
1. **Ingestion & Preprocessing**: Upload via Streamlit, enhanced locally via OpenCV (noise removal, deskewing).
2. **OCR**: PaddleOCR processes the enhanced image to extract handwritten text.
3. **LLM Structuring**: The cleaned text is sent to a local LLM (Qwen2.5:3B or Phi-3 Mini via Ollama) to extract entities into a JSON schema.
4. **Persistence & Export**: The JSON is saved to SQLite, and users can export the approved FIR as a PDF using ReportLab.

#### Key Design Decisions
- **OCR Engine Choice**: PaddleOCR selected over standard text extractors (like PyMuPDF) because the primary input is handwritten images.
  - *Trade-offs*: PaddleOCR is heavier and requires more CPU resources than simple text extraction.
  - *Recommendation*: Optimize PaddleOCR for CPU inference and consider caching OCR results during development.
- **LLM Pipeline**: Using Ollama for local inference.
  - *Trade-offs*: Small models might hallucinate IPC sections or struggle with complex JSON.
  - *Recommendation*: Use strict Pydantic validation and prompt engineering focused strictly on extraction (not legal advice).
- **Image Preprocessing**: Utilizing OpenCV before OCR.
  - *Trade-offs*: Adds processing time, but critically improves OCR accuracy on poor-quality mobile camera scans.

#### Alternatives Considered
- **Cloud Vision APIs (Google Vision / AWS Textract)**: *Rejected* due to the strict offline and data privacy requirements of police departments.
- **Direct LLM Vision (e.g., LLaVA)**: *Rejected* because running multi-modal LLMs on CPU is extremely slow compared to the pipeline of OCR + small text LLM.

## Risk & Gap Analysis

#### Requirement Ambiguities
- **Human Review Flow**: The requirement mentions "Officer only reviews and approves," but the project structure lacks an explicit state machine for Draft vs. Approved FIRs.
- **Language Support**: Handwritten complaints in India are often in regional languages. While marked as a "Future Enhancement," if the MVP only supports English, PaddleOCR and the LLM might completely fail on real-world test data.

#### Edge Cases
- **Unreadable Handwriting**: If OpenCV + PaddleOCR produce absolute garbage text, the LLM will hallucinate. There needs to be a threshold for OCR confidence to alert the officer that manual entry is required.
- **Multiple Entities**: Complaints often have multiple victims, multiple accused, or multiple stolen items. The JSON schema must handle arrays properly.

#### Technical Risks
- **CPU Resource Bottleneck**: Running OpenCV, PaddleOCR, and a local LLM sequentially on a standard laptop CPU might take 3-5 minutes per complaint. The Streamlit UI must handle long-running async tasks without timing out.
- **OCR Quality**: Handwriting recognition is notoriously difficult. The quality of the "Raw Complaint Text" is the biggest risk to the success of the LLM extraction.
