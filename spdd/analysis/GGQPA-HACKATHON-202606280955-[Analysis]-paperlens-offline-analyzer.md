# SPDD Analysis: PaperLens Offline Analyzer

## Original Business Requirement
The original business requirements are documented in `idea.md` and `pro.md`.
The goal is to build PaperLens, an offline-first, CPU-optimized AI application that converts unstructured research papers (PDFs) into structured JSON and stores it locally. The application must operate entirely without an internet connection, utilizing local Small Language Models (e.g., Qwen2.5 or Phi-3 Mini) via Ollama, PyMuPDF for document parsing, SQLite for storage, and Streamlit for the user interface.

#### New Concepts Required
- **Paper / Document**: The core entity representing an uploaded research paper. Contains metadata like title, authors, year, domain, summary, and the raw text/extracted JSON.
- **Document Chunk**: A logical segment of the paper (e.g., Abstract, Introduction, Methodology) to be processed by the LLM, necessitated by the context window limits of small local models.
- **ExtractedMetadata**: The structured JSON payload generated by the LLM, conforming to a strict schema containing keywords, datasets, algorithms, results, limitations, etc.
- **Keyword / Dataset / Algorithm**: Relational entities extracted from the paper to enable structured search and filtering in the database.

#### Key Business Rules
- **Total Offline Execution**: The entire pipeline (parsing, inference, database) must function without any internet access or cloud APIs.
- **JSON Validity Invariant**: The LLM output MUST be valid JSON matching the predefined schema. If validation fails, an automatic retry must be triggered.
- **Idempotency / Caching**: Uploaded files must be hashed. If a file is uploaded again, the system must serve the cached result rather than reprocessing.
- **Graceful Degradation**: The system must not crash on unreadable PDFs, OCR failures, model timeouts, or persistent invalid JSON, but instead display meaningful errors to the user.

## Strategic Approach

#### Solution Direction
The solution will be a modular Python application with a Streamlit frontend. The core pipeline follows: `PDF Upload -> PyMuPDF Text Extraction -> Text Normalization -> Section-based Chunking -> Ollama (Local LLM) Inference -> JSON Validation -> SQLite Persistence -> Streamlit Dashboard display/search`.

#### Key Design Decisions
- **LLM Choice & Sizing**: Use lightweight, quantized models (Qwen2.5:3B or Phi-3 Mini) via Ollama. 
  - *Trade-offs*: Smaller models are faster and run on CPU but are prone to instruction-following failures (e.g., breaking JSON format). 
  - *Recommendation*: Implement robust JSON parsing (e.g., regex extraction to find JSON blocks within markdown) and retry logic.
- **Chunking Strategy**: Split the paper into logical sections instead of passing the entire text.
  - *Trade-offs*: Prevents context window exhaustion and improves extraction accuracy on small models, but requires logic to aggregate extracted facts across multiple chunks.
  - *Recommendation*: Process the paper chunk by chunk, either extracting partial JSONs and merging them, or using a rolling summary approach.
- **Database Architecture**: Use SQLite for primary storage of documents and extracted relational entities (Keywords, Datasets).
  - *Trade-offs*: Simple, zero-configuration, and offline by default, but less suited for advanced vector similarity compared to FAISS/Chroma. 
  - *Recommendation*: Use SQLite for the MVP structured data requirements, keeping FAISS as an optional bonus feature.

#### Alternatives Considered
- **Full Document Inference**: Passing the entire 50-page PDF text directly to the LLM. *Rejected* because local small models on CPU will exhaust memory, hit context limits, and hallucinate or truncate the output.

## Risk & Gap Analysis

#### Requirement Ambiguities
- **Chunk Aggregation**: `pro.md` mentions chunking the paper (Abstract, Intro, etc.) and also extracting a single JSON structure. It's ambiguous how the system should merge the structured outputs from multiple chunks into a single JSON record.
- **Model Availability**: The requirement states "The only requirement should be that the local model has already been pulled via Ollama." We need to clarify how the app handles or prompts the user if the expected model is missing.

#### Edge Cases
- **Non-parseable / Scanned PDFs**: PyMuPDF will return empty or garbage text for image-based PDFs. If EasyOCR is optional/future scope, the MVP must detect empty text and gracefully reject the file or prompt for the OCR pipeline.
- **Streamlit Timeout / UI Freeze**: CPU inference for a full research paper could take several minutes. The UI must use asynchronous updates or progress bars to prevent the user from thinking the app has frozen.

#### Technical Risks
- **JSON Schema Hallucinations**: Small models frequently add extra keys or nest JSON incorrectly despite strict prompting. *Mitigation direction*: Use Pydantic for strict schema validation, and potentially utilize Ollama's structural output features (JSON mode) if supported by the chosen model.
- **CPU Performance**: Processing time may be unacceptably slow for end users. *Mitigation direction*: Stream LLM responses to the UI to provide immediate feedback, and aggressively cache parsed text and inference results.
